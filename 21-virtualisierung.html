Virtualisierung — VM, Hypervisor & Virtualisierungsarten

Bei der Virtualisierung simuliert Software das Vorhandensein von Hardware. Eine virtuelle Maschine (VM) verhält sich wie ein eigenständiger Rechner und läuft auf einem physischen Wirt. Der Hypervisor (Virtual Machine Monitor) betreibt mehrere VMs parallel. A

Virtuelle Maschine & Hypervisor

Bei der Virtualisierung wird mit Software das Vorhandensein von Hardware simuliert. Eine VM ist ein virtueller Rechner mit eigener (virtueller) Hardware und eigenem Betriebssystem (Gastsystem); sie läuft auf einem physischen Wirt/Host. Software und Hardware sind dabei logisch entkoppelt. Der Hypervisor (Virtual Machine Monitor, VMM) ermöglicht den parallelen Betrieb mehrerer VMs und partitioniert die Ressourcen (RAM, CPU, Speicher, Peripherie).

01Multiple ChoiceVMAuto

Was ist eine virtuelle Maschine (VM)?

  • Ein virtueller Rechner, der sich wie ein eigenständiger physischer Rechner mit eigener Hardware und eigenem Betriebssystem verhält.
  • Ein zweiter physischer Server im selben Rack.
  • Ein Cloud-Speicherdienst für Backups.
  • Ein Netzwerkprotokoll zur IP-Vergabe.
Lösung

Eine VM ist ein virtueller Rechner, der sich dem Benutzer gegenüber wie ein eigenständiger physischer Rechner (eigene Hardware, eigenes Betriebssystem) verhält. Man nennt sie auch virtuelles Gastsystem; sie läuft auf einem physischen Wirt/Host.

02Multiple ChoiceHypervisorAuto

Was ist ein Hypervisor?

  • Ein besonders schneller Prozessor für Server.
  • Software (Virtual Machine Monitor), die den parallelen Betrieb mehrerer VMs auf einem Wirt ermöglicht und die Hardwareressourcen partitioniert.
  • Ein Backup-Programm für virtuelle Festplatten.
  • Ein VPN-Gateway zur Standortkopplung.
Lösung

Der Hypervisor (auch Virtual Machine Monitor, VMM) ist die Abstraktionsschicht zwischen Wirt und Gastsystemen. Er ermöglicht den parallelen Betrieb mehrerer VMs und verwaltet/partitioniert die Hardwareressourcen (RAM, CPU, Speicherplatz, Peripherie).

B

Hypervisor-Typen 1 & 2

Der Typ-1- (Bare-Metal-)Hypervisor setzt direkt auf der Hardware auf — kein vorinstalliertes Betriebssystem. Der Typ-2- (Hosted-)Hypervisor läuft als Applikation auf einem vorhandenen Betriebssystem und nutzt dessen Treiber als Brücke zur Hardware.

04Multiple ChoiceTyp 1Auto

Was gilt für einen Typ-1- (Bare-Metal-)Hypervisor?

  • Er läuft als Anwendung auf einem vorhandenen Windows.
  • Er setzt direkt auf der Hardware auf — es ist kein vorinstalliertes Betriebssystem nötig.
  • Er ist grundsätzlich langsamer als ein Typ-2-Hypervisor.
  • Er funktioniert ausschließlich in der Cloud.
Lösung

Der Typ-1- (Bare-Metal-)Hypervisor setzt direkt auf der Hardware des Wirts auf und benötigt daher kein vorab installiertes Betriebssystem. Durch die „dünne" Schicht ist der Overhead gering → hohe Performance, Sicherheit, Stabilität.

05Multiple ChoiceTyp 2Auto

Welches ist ein Typ-2- (Hosted-)Hypervisor?

  • Microsoft Hyper-V
  • Oracle VirtualBox
  • VMware ESX
  • Citrix XenServer
Lösung

Oracle VirtualBox ist ein Typ-2- (Hosted-)Hypervisor (ebenso VMware Workstation, Microsoft Virtual-PC). Hyper-V, VMware ESX und Citrix XenServer sind Typ-1- (Bare-Metal-)Hypervisoren.

06Multiple ChoiceOverheadAuto

Warum hat ein Typ-2-Hypervisor tendenziell geringere Performance, dafür bessere Hardware-Kompatibilität?

  • Er greift über die Treiber des Host-Betriebssystems auf die Hardware zu (Brücke) — das erzeugt Overhead, aber das BS verwaltet die Treiber.
  • Er verschlüsselt jeden Hardware-Zugriff einzeln.
  • Er läuft direkt auf der Hardware ohne jede Zwischenschicht.
  • Er kann nur eine einzige VM gleichzeitig betreiben.
Lösung

Der Typ-2-Hypervisor nutzt die vom Host-BS bereitgestellten Gerätetreiber als Brücke. Der fehlende direkte Hardware-Zugriff erhöht den Overhead (geringere Performance), dafür ist die Hardware-Kompatibilität besser, weil das Betriebssystem die Treiber verwaltet.

C

Virtualisierungsarten

Man unterscheidet Server-, Desktop- und Applikationsvirtualisierung. Bei der Desktopvirtualisierung ist die VDI (Virtual Desktop Infrastructure) verbreitet — im Unterschied zu DaaS (Desktop as a Service) läuft sie on-premises im eigenen Rechenzentrum.

07Multiple ChoiceVDI vs. DaaSAuto

Worin unterscheiden sich VDI und DaaS?

  • VDI läuft in der Cloud, DaaS im eigenen Rechenzentrum.
  • VDI wird on-premises im eigenen Rechenzentrum gehostet, DaaS von einem Cloud-Anbieter.
  • Beide sind identisch, nur die Namen unterscheiden sich.
  • VDI benötigt keinen Server, DaaS schon.
Lösung

VDI hostet die virtuellen Desktops on-premises im unternehmenseigenen Rechenzentrum. DaaS ist ein Service, bei dem ein Cloud-Anbieter den virtuellen Desktop bereitstellt.

08Multiple ChoicepersistentAuto

Was kennzeichnet einen non-persistenten VDI-Desktop?

  • Er wird individuell angepasst und dauerhaft gespeichert.
  • Er wird beim Abmelden auf den Ursprungszustand zurückgesetzt (Fortschritt gelöscht).
  • Er liegt immer in der Cloud eines Anbieters.
  • Er benötigt keinen Server im Hintergrund.
Lösung

Bei einem non-persistenten Desktop greift der Nutzer auf einen Pool identischer Images zu; beim Abmelden wird der Desktop auf den Ursprungszustand zurückgesetzt — der Bearbeitungsfortschritt geht verloren. Ein persistenter Desktop bleibt dagegen individuell gespeichert.

09Multiple ChoiceSzenarioAuto

Ein Klassenzimmer wird von physischen PCs auf virtuelle Arbeitsplätze umgestellt. Welche Virtualisierungsart?

  • Servervirtualisierung
  • Desktopvirtualisierung (VDI)
  • Applikationsvirtualisierung
  • Netzwerkvirtualisierung
Lösung

Virtuelle Arbeitsplatzrechner gehören zur Desktopvirtualisierung; die Umrüstung eines Klassenzimmers auf virtuelle Desktops im eigenen Rechenzentrum ist ein klassischer VDI-Fall.

10Multiple ChoiceApplikationsvirt.Auto

Was beschreibt die Applikationsvirtualisierung?

  • Die Anwendung wird vom Betriebssystem isoliert und in einer eigenen Umgebung (Sandbox) bzw. zentral über einen Server bereitgestellt.
  • Ein physischer Server wird in mehrere virtuelle Server aufgeteilt.
  • Ein virtueller Desktop wird per Remote gestreamt.
  • Mehrere Standorte werden per VPN gekoppelt.
Lösung

Die Applikations-/Anwendungsvirtualisierung isoliert die Anwendung vom ausführenden Betriebssystem — sie läuft in einer eigenen Sandbox oder wird zentral auf einem Server bereitgestellt (nur eine Verknüpfung/.exe liegt auf dem Client). Tools: Bufferzone, Sandboxie; Anbieter: Citrix XenApp, VMware ThinApp, Microsoft App-V.

D

Vor- & Nachteile

Virtualisierung bringt vor allem Serverkonsolidierung und bessere Auslastung — hat aber auch Schattenseiten wie einen Single Point of Failure und Lizenz-/Komplexitätskosten.

12Multi-SelectVorteileAuto

Welche Aussagen nennen einen echten Vorteil der Virtualisierung? (Mehrfachauswahl)

  • Serverkonsolidierung — weniger Hardware und geringere Kosten.
  • Bessere Auslastung der Ressourcen.
  • Schnelles Bereitstellen/Klonen von VMs.
  • Isolation/Hochverfügbarkeit — der Absturz einer VM betrifft andere nicht.
  • Live-Migration: VMs ohne Downtime auf andere Hardware verschieben.
  • Sie beseitigt jeden Single Point of Failure.
  • Sie macht Software-Lizenzen überflüssig.
Lösung

Echte Vorteile: Serverkonsolidierung, bessere Auslastung, schnelles Klonen, Isolation/Hochverfügbarkeit, Live-Migration, einfacheres Backup/DR. Ein Single Point of Failure wird gerade nicht beseitigt (im Gegenteil), und Lizenzen bleiben nötig.

13Multi-SelectNachteileAuto

Welche Aussagen nennen einen echten Nachteil der Virtualisierung? (Mehrfachauswahl)

  • Single Point of Failure.
  • Lizenzkosten mancher Hypervisor-/Management-Tools.
  • Performance-Overhead (vor allem bei Typ 2).
  • Steigende Komplexität im Management.
  • Höherer Stromverbrauch durch mehr physische Server.
  • Es ist keinerlei Isolation der Systeme möglich.
Lösung

Nachteile: Single Point of Failure, Lizenzkosten, Performance-Overhead, steigende Komplexität. Der Stromverbrauch sinkt durch Serverkonsolidierung, und Isolation ist gerade ein Vorteil.

14Multi-SelectServervirt.Auto

Was trifft auf die Servervirtualisierung zu? (Mehrfachauswahl)

  • Ein physischer Server wird in mehrere virtuelle Server aufgeteilt.
  • Hardware und Software werden logisch entkoppelt.
  • Der Hypervisor teilt RAM/CPU/Speicher auf die Gastsysteme auf.
  • Jeder VM wird ein eigener physischer Server zugeteilt.
  • Sie funktioniert nur zusammen mit einem Cloud-Anbieter.
Lösung

Bei der Servervirtualisierung wird ein physischer Server per Virtualisierungssoftware in mehrere virtuelle Server aufgeteilt; Hardware und Software sind logisch entkoppelt, der Hypervisor partitioniert die Ressourcen. Ein eigener physischer Server je VM oder ein zwingender Cloud-Anbieter widerspricht genau dem Prinzip.

15SzenarioPrinzipmanuell

Für den Dateiserver und den ERP-Zugang der Schule ist eine Virtualisierung geplant. Erkläre das Prinzip der Servervirtualisierung.

Musterlösung
1. Ein physischer Server wird per Virtualisierungssoftware in mehrere isolierte virtuelle Maschinen aufgeteilt (z. B. eine VM für den Dateiserver, eine für ERP).
2. Software und Hardware werden logisch entkoppelt (Hardware-Virtualisierung).
3. Der Hypervisor verwaltet und partitioniert die Ressourcen (RAM, CPU, Speicher) auf die Gastsysteme.
4. Ergebnis: bessere Auslastung, Serverkonsolidierung (weniger Hardware/Strom) und Isolation der Dienste voneinander.
